# 06 — Living-Spider-System.md

# Portfolio V3

## Living Spider System

---

# Purpose

The spider is not an animation.

The spider is not a mascot.

The spider is not a feature.

The spider is a resident of the portfolio.

Its existence should make the digital workspace feel inhabited.

Visitors are not expected to notice the spider.

Those who do should feel as though they accidentally discovered something personal.

The spider should never interrupt the experience.

It should quietly belong to it.

---

# Core Philosophy

The spider should never ask for attention.

It should never perform.

It should never entertain.

It simply lives inside the portfolio.

Imagine noticing a tiny spider in the corner of a real room.

You smile.

You continue what you were doing.

That emotional response is the objective.

---

# Identity

The spider has no name.

No personality written into the UI.

No dialogue.

No achievements.

No collectibles.

No interaction prompts.

The visitor is observing wildlife.

Not interacting with a game character.

---

# Design Language

The spider should be extremely small.

Immediately recognizable.

Never exaggerated.

Avoid cartoon proportions.

Avoid realistic horror aesthetics.

Avoid detailed textures.

Instead,

design it as a clean silhouette.

Elegant.

Minimal.

Readable.

---

# Visibility Rules

The spider should never dominate the screen.

Maximum visual priority:

Very Low.

It should only occupy the visitor's peripheral vision.

If the spider becomes the focal point,

its implementation has failed.

---

# Spawn Philosophy

The spider never "appears."

It arrives.

It should always emerge from somewhere believable.

Examples

Behind the navigation bar.

Behind a glass panel.

From the edge of the browser.

From an existing web.

Behind typography.

Near the monitor frame.

Never materialize in empty space.

---

# Spawn Timing

The spider should feel unpredictable.

Never appear immediately after page load.

Never appear at fixed intervals.

Every appearance should feel accidental.

Time between appearances should vary naturally.

The visitor should never learn a pattern.

---

# Cooldown System

After completing an appearance,

the spider should disappear for a significant period.

Repeated encounters reduce emotional impact.

Rarity creates delight.

---

# Path Selection

The spider should never travel in straight lines.

Movement should include

tiny pauses,

slight hesitation,

minor direction changes,

small corrections.

Perfect movement feels artificial.

Imperfect movement feels alive.

---

# Walking Behaviour

Walking speed should vary slightly.

The spider occasionally pauses.

Looks around.

Continues walking.

Small unpredictability creates realism.

---

# Silk Behaviour

Whenever possible,

the spider should use silk.

Silk should feel incredibly thin.

Almost invisible.

The silk should remain under gentle tension.

Never behave like rope.

Never bounce dramatically.

Never glow.

---

# Climbing Behaviour

When climbing,

movement should be slower than walking.

Gravity should be implied.

The spider should occasionally pause while climbing.

Just like a real creature.

---

# Escape Behaviour

If the cursor approaches too closely,

the spider reacts.

Not instantly.

It notices first.

Pauses.

Changes direction.

Finds safety.

Disappears.

It should never panic.

It should never sprint unrealistically.

---

# Curiosity Behaviour

Sometimes,

instead of escaping,

the spider may simply stop.

Remain still.

Watch.

Then continue walking.

This tiny uncertainty makes it believable.

---

# Environmental Awareness

The spider understands the workspace.

It avoids

buttons,

important text,

active forms,

critical UI.

It prefers

edges,

corners,

glass,

frames,

empty space.

The spider respects the visitor's work.

---

# Night Behaviour

At night,

the spider becomes slightly more active.

Not dramatically.

Only enough that observant visitors notice.

The workspace feels naturally inhabited after dark.

---

# Weather Awareness

During rain,

the spider should appear less frequently.

During lightning,

it should remain hidden.

During calm weather,

normal behaviour resumes.

The spider belongs to the world.

The world influences it.

---

# Reading Awareness

While the visitor is reading,

the spider should avoid entering the reading area.

Reading remains sacred.

Nothing should compete with text.

---

# Scroll Awareness

The spider should never continue walking during rapid scrolling.

Instead,

it should disappear naturally.

Once scrolling settles,

future appearances become possible again.

---

# Session Awareness

The spider remembers nothing.

Each visit begins fresh.

Every visitor experiences unique encounters.

No persistence.

No tracking.

No behavioural profiling.

Privacy is respected.

---

# Curiosity Reward

Very observant visitors may notice

that the spider occasionally returns to the same hidden web.

No explanation.

No reward.

No message.

The world simply feels consistent.

---

# Hidden Console Easter Egg

If a visitor repeatedly follows the spider until it escapes,

a single console message may appear.

> You scared him away.

Nothing else.

No achievement.

No badge.

No popup.

Only developers who intentionally inspect the console discover it.

---

# Permanent Web

Somewhere within the portfolio,

one tiny spider web permanently exists.

It is subtle.

Almost invisible.

Most visitors will never notice it.

Those who do should wonder

whether it has always been there.

Never highlight it.

Never animate it dramatically.

---

# Animation Principles

The spider should obey the Motion Language.

Acceleration.

Momentum.

Weight.

Natural stopping.

No robotic movement.

No linear paths.

No exaggerated anticipation.

Everything should remain believable.

---

# Accessibility

Respect reduced motion.

When reduced motion is enabled,

the spider system is disabled.

The permanent web may remain.

Nothing else.

Accessibility always has priority.

---

# Performance

Only one spider may exist at any time.

Never create multiple spiders.

Never run continuous pathfinding.

Behaviour should be lightweight.

GPU friendly.

Battery friendly.

The spider should consume almost no resources while inactive.

---

# Failure Behaviour

If the spider system fails,

nothing else should break.

The portfolio should simply continue without it.

The visitor should never know it existed.

---

# Things the Spider Must NEVER Do

Never jump-scare.

Never cover text.

Never crawl over buttons.

Never trigger notifications.

Never speak.

Never wave.

Never dance.

Never become clickable.

Never become a mini-game.

Never become part of navigation.

Never become the hero of the experience.

---

# Definition of Success

The perfect implementation creates stories.

A visitor should someday tell a friend:

> "I swear I saw a tiny spider walking across the page."

And the friend should reply:

> "Wait... really? I never noticed."

That uncertainty is the highest compliment.

The spider should not be remembered because it was obvious.

It should be remembered because it felt real.

---

# Final Principle

The portfolio belongs to the visitor while they are exploring it.

The spider simply happens to live there.

Treat it with the same respect you would give any living thing sharing a quiet workspace.

If visitors forget about the implementation but remember the feeling of discovering something unexpectedly human,

the Living Spider System has achieved its purpose.
